# Astro 6 Upgrade Notes

Upgraded from Astro 5.17.1 to Astro 6.3.7. The migration was largely
mechanical; this file captures the things that downstream consumers and
content authors need to be aware of.

## Things that changed in this repo

### 1. `z` is no longer re-exported from `astro:content`

Astro v6 removed `astro:schema` and stopped re-exporting `z` from
`astro:content`. Affected files import `z` from `astro/zod` instead.

```diff
- import { defineCollection, z } from 'astro:content'
+ import { defineCollection } from 'astro:content'
+ import { z } from 'astro/zod'
```

If anyone copy-pastes the old import pattern into a new collection, it
will fail. The error is loud, so it's safe.

### 2. `@/config` virtual-module aliasing is now done in `resolveId`

Vite 7 (bundled with Astro 6) is stricter about `resolve.alias`
replacements that point at a Vite *virtual module specifier* rather than
a real path. Under the old alias scheme, the content-collections and
prerender build environments would fall through the alias and pull in
the on-disk `src/config.ts`, which calls `readFileSync` at module load.
That file then gets bundled into the prerender chunk, and at runtime the
`new URL('../default-config.yaml', import.meta.url)` lookup resolves to
`dist/.prerender/default-config.yaml` — a path that does not exist —
which crashes the build with `ENOENT`.

The fix moves the `@/config` → virtual module redirection into the
plugin's own `resolveId` hook (with `enforce: 'pre'`) so it runs ahead
of alias resolution and applies uniformly to every build environment.
See `integration.ts`.

### 3. Tag-route param encoding is now minimal

In Astro 5 we pre-encoded the entire tag with `encodeURIComponent` in
`getStaticPaths` to support tags containing slashes (e.g. `CI/CD`). In
Astro 6 the router normalises params more aggressively, so a pre-encoded
space (`Software%20Engineering`) no longer matches the same incoming
URL — the build fails with `NoMatchingStaticPathFound: /tags/Software Engineering`.

We now encode *only* slashes, which is the one character the router
genuinely cannot keep inside a single dynamic segment. Everything else
(spaces, CJK, punctuation) is handed to Astro raw and the framework
encodes it for us at link-emit time and decodes it again at match time.

```diff
- params: { ..., tag: encodeURIComponent(tag) },
+ params: { ..., tag: tag.replace(/\//g, '%2F') },
```

If a future tag needs additional special handling (e.g. backslashes),
extend the replace pattern. The `getTagPath` helper still does a full
`encodeURIComponent` for link generation, which is correct.

### 4. `@qwik.dev/partytown` patch was dropped

`@astrojs/partytown@2.1.7` upgraded its bundled partytown from 0.11.2 to
0.13.2. Our old patch (renaming the `t` global to avoid a collision)
targets the 0.11.2 file and no longer applies. We removed the patch
entry from `package.json`. If a name collision resurfaces, recreate the
patch against the new 0.13.2 source. So far the standalone build runs
clean without it.

## Things that did *not* need fixing but might still bite

### Markdown heading IDs

Astro 6 changed the heading-id algorithm: trailing hyphens generated by
stripping special characters are no longer trimmed. A heading like
`## \`<Picture />\`` now produces `id="picture-"` instead of
`id="picture"`.

We do **not** patch this — our content's headings already round-trip
cleanly through the new algorithm in a spot-check of the built output —
but any link inside a post that hard-codes a `#fragment` to a heading
with special characters may now drift by one hyphen. If you notice a
broken in-page anchor after the upgrade, this is the most likely cause:
rebuild and inspect the rendered `<h2 id="…">` in `dist/`.

### `redirectToDefaultLocale` default flipped

The default for `i18n.routing.redirectToDefaultLocale` changed from
`true` to `false` in Astro 6. We were already setting this explicitly
(`integration.ts` registers `routing: { prefixDefaultLocale: false,
redirectToDefaultLocale: false, fallbackType: 'redirect' }`), so the
runtime behaviour matches v5. No action needed; the explicit value is
the safety net.

### Node 22 is now required

Astro 6 dropped Node 18 and 20. Local dev is on 22.16.x; if CI is still
pinned to an older line, it will fail.

### Zod 4 deprecation warnings

`astro check` reports deprecation hints for `.passthrough()` in
`src/content.config.ts`. The method still works in the bundled Zod 4 but
will be removed eventually. The schemas are content frontmatter
contracts — migrating them away from `.passthrough()` is a separate,
content-shaped task and not required to ship Astro 6.
